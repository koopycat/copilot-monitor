Local-only proxy

Know exactly what GitHub Copilot is doing on your machine.

A local HTTP reverse proxy that records per-request metadata, token counts, latency, and an estimated AI-credit cost. Stored in SQLite on your machine. No cloud. No telemetry. No prompts, completions, source code, or auth headers are ever written to disk.

Why it exists

GitHub Copilot picks models automatically, charges AI credits invisibly, and exposes usage summaries only on github.com with a 24-hour delay. You don't know which model handled your last prompt, whether you got a cache hit, or what that refactor session actually cost you.

Copilot Monitor gives you the raw numbers from your own machine: per-model token counts, latency, estimated cost, and 30-minute session groupings. Local dashboard with a period selector (today / yesterday / 7d / 30d / 90d / 365d) and a JSON API for your own scripts.

How it works

A reverse proxy sits between VSCode and the Copilot API, capturing metadata on the way through. Output goes to a local SQLite file and a dashboard on loopback.

request → copilot-monitor → response  ·  metadata → SQLite  ·  dashboard reads SQLite

What is and isn't stored

The privacy contract is enforced by the schema, not by policy. No prompt, completion, source code, or auth material is ever written to disk.

Stored

  • Timestamp, endpoint, HTTP status
  • Model name (when known)
  • Latency (ms)
  • Token counts: input, cached input, cache write, output
  • Project label (optional)
  • Session grouping (derived)

Never stored

  • Prompt text
  • Completion text
  • Source code or repository paths
  • Auth headers, cookies, API keys
  • Anything from request or response bodies
  • Loopback only. The proxy binds to 127.0.0.1 and the dashboard binds to 127.0.0.1. Nothing is reachable from another machine.
  • No external network calls. The binary never phones home, never updates itself, and never sends analytics. Verified by reading the source.
  • Local file only. SQLite lives at ~/.local/share/copilot-monitor/store.db on your machine. If you delete the file, every trace is gone.

Is this for you?

Use it if any of these sound like you.

You use Copilot every day

$ copilot-monitor cost --since 30d

You want to know which models you're actually calling, and what your prompts cost. Not a corporate dashboard, your own numbers on your own machine.

You're tuning prompts for cache hit rate

cache % per model, per session

Copilot Monitor breaks cached-input versus uncached tokens down per session, so you can see which prompt patterns actually pay off.

You don't trust cloud-side telemetry

binds 127.0.0.1 only

Read the schema, inspect the source, verify nothing leaves your machine. The privacy contract is enforced by the schema, not by a privacy policy.

You're piping usage into your own tooling

--json on every command

Every endpoint returns JSON. Every CLI command has --json. Queries are SQL against a SQLite file you own.

Build from source

Clone, build, run. The devenv shell provides the Go toolchain.

# clone and enter the dev shell
git clone https://github.com/koopycat/copilot-monitor
cd copilot-monitor
devenv shell

# build, then run the proxy + dashboard
just build
./copilot-monitor run &
./copilot-monitor serve

# point VSCode at the proxy
./copilot-monitor configure-vscode >> ~/.config/Code/User/settings.json
# reload VSCode and use Copilot normally

Open http://127.0.0.1:7734 for the dashboard.
