Get your first capture

Install the binary, route one real request through the local proxy, then check the dashboard. VS Code needs a one-time advanced setting and reload; pi needs an OpenRouter API key.

Install with Homebrew macOS · apple silicon

brew tap koopycat/copilot-monitor
brew trust koopycat/copilot-monitor
brew install copilot-monitor

Puts copilot-monitor on your PATH. The formula is Apple Silicon only; on Intel Mac use the archive route.

Install from an archive

tar -xzf copilot-monitor-*.tar.gz
sudo mv copilot-monitor /usr/local/bin/

After downloading a build above. On Windows, extract the .zip and put copilot-monitor.exe on your PATH.

VS Code + GitHub Copilot

All traffic is forwarded to the configured --upstream host.

one-time setting
  1. 1

    Start the proxy and dashboard

    copilot-monitor run --dashboard --upstream api.githubcopilot.com
  2. 2

    Point Copilot at the proxy

    Open Preferences: Open User Settings (JSON) and add:

    {
  "github.copilot.advanced": {
    "debug.overrideCapiUrl": "http://127.0.0.1:7733"
  }
}

    To revert later, remove the debug.overrideCapiUrl entry and reload the window.

  3. 3

    Reload and send one Copilot chat

    Run Developer: Reload Window, use Copilot normally, then open 127.0.0.1:7734 once the dashboard is running to confirm the capture.

    If the dashboard is unavailable or empty, run:

    copilot-monitor doctor --upstream api.githubcopilot.com

pi + OpenRouter

pi ships with an OpenRouter provider, so an API key and a base URL override are all you need. Detailed guide

base URL override
  1. 1

    Start the proxy and dashboard

    copilot-monitor run --dashboard --upstream openrouter.ai
  2. 2

    Point pi's OpenRouter provider at the proxy

    Add to ~/.pi/agent/models.json. The built-in model catalog is kept: 

    {
  "providers": {
    "openrouter": { "baseUrl": "http://127.0.0.1:7733/api/v1" }
  }
}

    Authenticate with export OPENROUTER_API_KEY=... or pi's auth.json.

  3. 3

    Send one pi request

    With any OpenRouter model selected (pick one with /model):

    pi -p 'Reply OK'

    Open 127.0.0.1:7734 once the dashboard is running to inspect the capture.

    If the dashboard is unavailable or empty, run:

    copilot-monitor doctor --upstream openrouter.ai

    Expected output when the local setup is healthy:

    Result: ready

Any OpenAI-compatible client works the same way: point its base URL at the proxy, keeping the path prefix the real API expects (http://127.0.0.1:7733/api/v1 for OpenRouter, http://127.0.0.1:7733/v1 for OpenAI), and start the proxy with --upstream set to that API's host. The proxy forwards the request path unchanged.

Why it exists

AI coding tools can hide the request-level details that explain a slow session, a changing model mix, or unexpectedly high token use. Account and organization dashboards answer a different question than the one a developer has while working locally.

Copilot Monitor records the traffic you choose to proxy: per-model token counts, HTTP status, latency, estimated cost, and 30-minute session groupings. It gives one developer a fast local record without deploying a shared control plane.

It is not another multi-provider gateway. Gateways manage routing, credentials, quotas, and team-wide controls; Copilot Monitor complements them as a local instrument for seeing the traffic from your own tools.

How it works

A reverse proxy sits between your tools and LLM APIs, capturing metadata on the way through. Output goes to a local SQLite file and a dashboard on loopback.

request → copilot-monitor → response  ·  metadata → SQLite  ·  dashboard reads SQLite

What you get

$

Per-model cost estimates, not a bill

Embedded per-token USD rates turn captured usage into a directional estimate, broken down by input, cached input, cache write, and output tokens. It does not reconcile plan allocation, included credits, discounts, or invoices. Fallback and not-billed rows are tagged.

$ ./copilot-monitor cost --since 7d --json | jq .total_usd
3.47
⌨

CLI + JSON API

Core reports (stats, cost, today, sessions, live, and inspect) support machine-readable JSON. export writes CSV, and the local API exposes the same captured data for your own tooling.

⏱

Live session view

Requests grouped by a 30-minute inactivity gap. See the model mix, cost, and duration of the session you're in right now.

📅

Period drill-down

Today / yesterday / 7d / 30d / 90d / 365d. Every section of the dashboard filters to the same window.

🔒

Local monitoring by design

Binds to 127.0.0.1. SQLite at ~/.local/share/copilot-monitor/store.db. No monitoring cloud, account, analytics, or phone-home.

✓

Inspectable

Single Go binary. Schema in internal/store/schema.sql. Read the source, audit the captures, trust what you see.

What is and isn't stored

Normal capture is metadata-only: no prompt, completion, source code, or auth material is written to the SQLite database. Raw debugging is an explicit exception and is marked as sensitive.

Recorded on the record

  • Timestamp, endpoint, HTTP status
  • Model name (when known)
  • Latency (ms)
  • Token counts: input, cached, cache write, output
  • Project label (optional)
  • Session grouping (derived)
Redacted

Not stored by normal capture blacked out

  • prompt text
  • completion text
  • source code / repo paths
  • auth headers, cookies, keys
  • request / response bodies
  • Loopback by default. The proxy and dashboard bind to 127.0.0.1 unless you explicitly change their listen address.
  • No monitoring network calls. The binary never phones home, updates itself, or sends analytics. It does forward the API request to the upstream host you explicitly configured.
  • Local database. Normal capture lives at ~/.local/share/copilot-monitor/store.db on your machine. Opt-in raw debug logs are separate sensitive files.

Is this for you?

Use it if any of these sound like you.

FILE 01

You use Copilot every day

$ copilot-monitor cost --since 30d

You want to know which models you're actually calling and their estimated token-rate cost. Not a corporate dashboard. Your own numbers, on your own machine.

FILE 02

You're tuning prompts for cache hit rate

cache % per model, per session

Copilot Monitor breaks cached-input versus uncached tokens down per session, so you can see which prompt patterns actually pay off.

FILE 03

You don't trust cloud-side telemetry

binds 127.0.0.1 only

Read the schema and inspect the source. Monitoring data stays on your machine; only the upstream API requests you explicitly proxy leave it.

FILE 04

You're piping usage into your own tooling

JSON reports + local API

Reporting commands that support --json and the local API are ready for your own tooling. Export is CSV. Queries are SQL against a SQLite file you own.
